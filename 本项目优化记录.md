最后更新：2026-07-10

本文记录 `codex-trace-viewer` 已完成和后续可继续推进的性能、交互优化。每个条目按同一口径写：现象、原因、方案、涉及文件、效果和验证。后续新增优化继续往下追加，不把某一次优化写成整篇主题。

## 1. 分割线拖动卡顿优化

### 1.1 现象

拖动左右面板分割线，或拖动 segment 详情面板分割线时，内容区会跟着持续重排。trace 内容多、卡片多、代码块多时，拖动过程容易出现卡顿。

### 1.2 原因

如果在每一次 `pointermove` 里直接修改真实布局尺寸，浏览器需要连续重新计算：

- 左侧 session 面板宽度。
- 右侧内容区宽度。
- segment 详情面板高度。
- 消息列表 padding、卡片布局和代码块排版。

拖动事件频率很高，真实布局每帧都变，就容易出现 layout / paint 压力。

### 1.3 方案

改成“拖动时只移动拖影，松手后再更新真实布局”。

- `pointerdown` 时记录起始尺寸和鼠标位置。
- 拖动过程中创建轻量的 `.splitterGhost`。
- `pointermove` 只更新 ghost 的 `transform`，并通过 `requestAnimationFrame` 合并高频事件。
- `pointerup` / `pointercancel` 时再一次性写入真实 CSS 变量：
  - `--session-pane-width`
  - `--segment-detail-height`
- 使用 pointer capture 和 window 级 pointer 事件，保证鼠标拖出手柄范围后也能正常结束拖动。

### 1.4 涉及文件

- `codex-trace-viewer/public/app.js`
  - `initSplitters()`
  - `initPointerResize()`
  - `createSplitterGhost()`
- `codex-trace-viewer/public/style.css`
  - `.splitter`
  - `.splitterGhost`
  - `.splitterGhost-x`
  - `.splitterGhost-y`
  - `--session-pane-width`
  - `--segment-detail-height`

### 1.5 效果

拖动过程中不再反复重排复杂内容，只移动一根轻量拖影线。真实布局只在松手时更新一次，明显减少拖动时的 layout thrash。

### 1.6 状态

## 2. turn / item 单击快速跳转优化

### 2.1 现象

点击左侧 turn，或点击某个 turn 内的 item 后，右侧内容会从当前位置一路滚到目标位置。距离远时，用户会看到明显滚动动画，体感是慢、晃眼、不够干脆。

### 2.2 交互约束

本次不改产品交互，只改内部实现。

- 单击 turn 仍然跳转。
- 单击 item 仍然跳转。
- 不增加双击、确认按钮、延迟触发等新交互。
- 目标是接近 ChatGPT Codex Desktop 那种“点完立刻到”的体感。

### 2.3 原因

原实现里，“选中”和“滚动”绑得太紧。

- `selectTurn()` 在部分场景会清空 `conversationViewSignature` 并调用 `render()`，导致右侧内容重新构建。
- 跳转使用 `scrollIntoView({ behavior: "smooth" })`，长距离跳转时浏览器会播放完整滚动动画。
- 选中态更新、详情面板更新、右侧滚动定位混在一起，容易形成“先重建，再慢慢滚过去”的体感。

### 2.4 参考思路

ChatGPT Codex Desktop 的跳转体感不是把滚动动画调快，而是尽量不展示长距离滚动过程：

- 先确定目标 turn / item 的位置。
- 直接把滚动容器放到目标附近。
- 用选中态或短暂高亮告诉用户目标在哪里。

完整 Codex Desktop 还涉及 turn 级虚拟化、可见范围渲染、高度测量等机制。本项目这次不照搬整套虚拟化，只先实现对体感最关键的部分：单击后 instant 定位。

### 2.5 方案

改成“单击仍跳转，但内部不再 smooth scroll”。

- 新增 `jumpToConversationElement()`，只操作右侧 `.conversationMessages` 滚动容器。
- 通过 `getBoundingClientRect()` 计算目标元素和滚动容器的相对位置。
- 直接设置 `container.scrollTop`，不再使用 `scrollIntoView()`。
- `scrollToTurnLifecycle()` 改为调用 instant 定位函数。
- turn 内 item 点击时，更新选中态和详情面板后，直接把目标 item 定位到容器中部。
- 如果目标 turn 已在当前右侧 DOM 中，`selectTurn()` 走快速路径：
  - 不清空 `conversationViewSignature`。
  - 不全量重建右侧内容。
  - 只更新 turn 选中态、segment rail、详情面板和目标位置。
- 增加 `jumpFlash` 短暂高亮，弥补没有滚动动画后的定位感。

### 2.6 涉及文件

  - `conversationScrollDelta()`
  - `setConversationScrollTop()`
  - `flashJumpTarget()`
  - `jumpToConversationElement()`
  - `turnLifecycleElement()`
  - `scrollToTurnLifecycle()`
  - `findThreadTurnInSession()`
  - `selectTurn()`
  - segment rail item 点击逻辑
  - token 表跳转逻辑
- `codex-trace-viewer/public/style.css`
  - `.jumpFlash`
  - `@keyframes jumpFlashPulse`

### 2.7 效果

优化前：

- 单击 turn 后，右侧可能先重建，再 smooth 滚动。
- 长距离跳转会看到明显滚动过程。
- 内容多时像是在“滚过去”，不够利落。

优化后：

- 单击 turn / item 仍然跳转。
- 跳转变成 instant 定位。
- 同线程 turn 切换时，目标已在 DOM 中就不全量重建右侧内容。
- 目标位置有短暂高亮反馈。
- 体感接近 Codex Desktop 的快速跳转方式。

## 3. 后续可追加优化

这些不是本次单击快速跳转的必要前提，但如果 trace 数据继续变大，可以继续拆成独立条目推进。

### 3.1 turn 级虚拟化

只渲染当前可见区域附近的 turn，远处 turn 用占位高度承接。适合解决超大 trace 下 DOM 节点太多的问题。

### 3.2 turn 高度缓存

记录每个 turn 渲染后的高度。后续跳转时可以更快估算目标位置，减少重新测量成本。

### 3.3 滚动反向同步左侧 active

右侧内容由用户手动滚动时，用 `IntersectionObserver` 或类似机制反向更新左侧当前 active turn。这样左侧导轨和右侧视口保持同步。

### 3.4 大 trace 分段加载

如果单个 trace 文件继续膨胀，可以考虑按 thread / turn 分段解析和加载，避免首屏一次性处理全部数据。

### 3.5 布局尺寸持久化

把用户调整后的 session 面板宽度、segment 详情面板高度写入本地存储。下次打开时恢复上次布局，减少重复调整。
