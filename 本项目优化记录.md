本文记录 `codex-trace-viewer` 已完成和后续可继续推进的性能、交互优化。每个条目按同一口径写：现象、原因、方案、涉及文件、效果和验证。后续新增优化继续往下追加，不把某一次优化写成整篇主题。

## 1. 分割线拖动卡顿优化

### 1.1 现象

拖动左右面板分割线，或拖动 segment 详情面板分割线时，内容区会跟着持续重排。trace 内容多、卡片多、代码块多时，拖动过程容易出现卡顿。

### 1.2 原因

如果在每一次 `pointermove` 里直接修改真实布局尺寸，浏览器需要连续重新计算：

- 左侧 session 面板宽度。
- 右侧内容区宽度。
- segment 详情面板高度。
- 消息列表 padding、卡片布局和代码块排版。

拖动事件频率很高，真实布局每帧都变，就容易出现 layout / paint 压力。

### 1.3 方案

改成“拖动时只移动拖影，松手后再更新真实布局”。

- `pointerdown` 时记录起始尺寸和鼠标位置。
- 拖动过程中创建轻量的 `.splitterGhost`。
- `pointermove` 只更新 ghost 的 `transform`，并通过 `requestAnimationFrame` 合并高频事件。
- `pointerup` / `pointercancel` 时再一次性写入真实 CSS 变量：
  - `--session-pane-width`
  - `--segment-detail-height`
- 使用 pointer capture 和 window 级 pointer 事件，保证鼠标拖出手柄范围后也能正常结束拖动。

### 1.4 涉及文件

- `codex-trace-viewer/public/app.js`
  - `initSplitters()`
  - `initPointerResize()`
  - `createSplitterGhost()`
- `codex-trace-viewer/public/style.css`
  - `.splitter`
  - `.splitterGhost`
  - `.splitterGhost-x`
  - `.splitterGhost-y`
  - `--session-pane-width`
  - `--segment-detail-height`

### 1.5 效果

拖动过程中不再反复重排复杂内容，只移动一根轻量拖影线。真实布局只在松手时更新一次，明显减少拖动时的 layout thrash。

### 1.6 状态

## 2. turn / item 单击快速跳转优化

### 2.1 现象

点击左侧 turn，或点击某个 turn 内的 item 后，右侧内容会从当前位置一路滚到目标位置。距离远时，用户会看到明显滚动动画，体感是慢、晃眼、不够干脆。

### 2.2 交互约束

本次不改产品交互，只改内部实现。

- 单击 turn 仍然跳转。
- 单击 item 仍然跳转。
- 不增加双击、确认按钮、延迟触发等新交互。
- 目标是接近 ChatGPT Codex Desktop 那种“点完立刻到”的体感。

### 2.3 原因

原实现里，“选中”和“滚动”绑得太紧。

- `selectTurn()` 在部分场景会清空 `conversationViewSignature` 并调用 `render()`，导致右侧内容重新构建。
- 跳转使用 `scrollIntoView({ behavior: "smooth" })`，长距离跳转时浏览器会播放完整滚动动画。
- 选中态更新、详情面板更新、右侧滚动定位混在一起，容易形成“先重建，再慢慢滚过去”的体感。

### 2.4 参考思路

ChatGPT Codex Desktop 的跳转体感不是把滚动动画调快，而是尽量不展示长距离滚动过程：

已移除不透明的旧图片资源。

- 先确定目标 turn / item 的位置。
- 直接把滚动容器放到目标附近。
- 用选中态或短暂高亮告诉用户目标在哪里。

完整 Codex Desktop 还涉及 turn 级虚拟化、可见范围渲染、高度测量等机制。本项目这次不照搬整套虚拟化，只先实现对体感最关键的部分：单击后 instant 定位。

### 2.5 方案

改成“单击仍跳转，但内部不再 smooth scroll”。

- 新增 `jumpToConversationElement()`，只操作右侧 `.conversationMessages` 滚动容器。
- 通过 `getBoundingClientRect()` 计算目标元素和滚动容器的相对位置。
- 直接设置 `container.scrollTop`，不再使用 `scrollIntoView()`。
- `scrollToTurnLifecycle()` 改为调用 instant 定位函数。
- turn 内 item 点击时，更新选中态和详情面板后，直接把目标 item 定位到容器中部。
- 如果目标 turn 已在当前右侧 DOM 中，`selectTurn()` 走快速路径：
  - 不清空 `conversationViewSignature`。
  - 不全量重建右侧内容。
  - 只更新 turn 选中态、segment rail、详情面板和目标位置。
- 增加 `jumpFlash` 短暂高亮，弥补没有滚动动画后的定位感。

### 2.6 涉及文件

- `codex-trace-viewer/public/app.js`
  - `jumpFlashTimers`
  - `conversationScrollDelta()`
  - `setConversationScrollTop()`
  - `flashJumpTarget()`
  - `jumpToConversationElement()`
  - `turnLifecycleElement()`
  - `scrollToTurnLifecycle()`
  - `findThreadTurnInSession()`
  - `selectTurn()`
  - segment rail item 点击逻辑
  - token 表跳转逻辑
- `codex-trace-viewer/public/style.css`
  - `.jumpFlash`
  - `@keyframes jumpFlashPulse`

### 2.7 效果

优化前：

- 单击 turn 后，右侧可能先重建，再 smooth 滚动。
- 长距离跳转会看到明显滚动过程。
- 内容多时像是在“滚过去”，不够利落。

优化后：

- 单击 turn / item 仍然跳转。
- 跳转变成 instant 定位。
- 同线程 turn 切换时，目标已在 DOM 中就不全量重建右侧内容。
- 目标位置有短暂高亮反馈。
- 体感接近 Codex Desktop 的快速跳转方式。

## 3. 后续可追加优化

这些不是本次单击快速跳转的必要前提，但如果 trace 数据继续变大，可以继续拆成独立条目推进。

### 3.1 turn 级虚拟化

只渲染当前可见区域附近的 turn，远处 turn 用占位高度承接。适合解决超大 trace 下 DOM 节点太多的问题。

### 3.2 turn 高度缓存

记录每个 turn 渲染后的高度。后续跳转时可以更快估算目标位置，减少重新测量成本。

### 3.3 滚动反向同步左侧 active

右侧内容由用户手动滚动时，用 `IntersectionObserver` 或类似机制反向更新左侧当前 active turn。这样左侧导轨和右侧视口保持同步。

### 3.4 大 trace 分段加载

如果单个 trace 文件继续膨胀，可以考虑按 thread / turn 分段解析和加载，避免首屏一次性处理全部数据。

### 3.5 布局尺寸持久化

把用户调整后的 session 面板宽度、segment 详情面板高度写入本地存储。下次打开时恢复上次布局，减少重复调整。

## 4. 高频 Trace 下全量会话刷新卡顿优化

### 4.1 现象

Trace 持续高频输入时，页面可能出现以下问题：

- 浏览器渲染进程内存上升到 1.6-2 GB。
- 页面滚动、点击 turn / item、打开详情时偶发停顿。
- 刷新页面后 `Connecting...` 持续较长时间。
- `/api/conversations` 单次响应可达到几十 MB，并需要十几秒才能完成。
- 同期很小的 `/api/storage` 请求也会被延迟，说明服务端主线程存在排队或阻塞。

### 4.2 已确认原因

当前刷新链路是：

```text
每条 SSE Trace 事件
  -> scheduleConversationRefresh()
  -> 250ms 后请求 /api/conversations
  -> 下载和解析完整会话模型
  -> 刷新 session 列表和当前会话内容
```

其中存在四个相互放大的问题：

1. `conversationRefreshScheduled` 在请求开始时就被清零，而不是在请求结束后清零。持续有 Trace 时，上一个请求尚未完成，250ms 后可以再次启动新请求。
2. `/api/conversations` 每次遍历全部 session、thread、turn 和 block，并同步执行大体积 `JSON.stringify()`。
3. 前端每次都下载和 `res.json()` 解析完整模型，没有利用 `conversationVersion` 跳过无变化响应。
4. 首次加载必须等待完整 `/api/conversations`，完成后才调用 `connectSse()`，因此大响应会直接延长 `Connecting...`。

内存占用不是最初触发原因，但多个大响应、JSON 字符串、解析对象和 DOM 同时存在，会增加垃圾回收压力并进一步放大卡顿。

### 4.3 第一阶段：禁止重入和无效刷新

目标是用低风险改动先停止请求堆积。

- 同一时间只允许一个 `/api/conversations` 请求。
- 请求期间收到新事件，只设置 dirty 标记。
- 当前请求结束后，如果 dirty，再补一次刷新，不积累多次任务。
- 活跃页面限制刷新频率；页面不可见时显著降低频率。
- 请求携带当前 `conversationVersion`，服务端版本未变化时返回轻量 `unchanged` 结果。
- 只有模型版本变化时才调用 `renderSessionsAndConversation()`。

### 4.4 第二阶段：摘要和详情拆分

目标是让首屏不再下载几十 MB 的全部历史详情。

- `/api/conversations/summary` 只返回 session / thread 摘要、数量、标题和更新时间。
- `/api/conversations/thread?id=...` 只返回当前选中 thread 的 turns 和 blocks。
- 完整命令输出、diff、tool result 和 raw events 在点击详情时按需请求。
- 首次加载先建立 SSE 和显示摘要，再异步加载当前 thread。
- 记住上次选中的 session / thread，刷新页面后优先恢复该范围。

### 4.5 第三阶段：增量更新和局部渲染

目标是让每条 Trace 只更新真正变化的范围。

- SSE 携带变化的 sessionId、threadId、turnId 和 itemId。
- 增加基于版本号的 changes 接口，只返回指定版本后的变化。
- 新 item 只追加对应卡片；状态变化只更新对应状态和耗时。
- session 列表只更新发生变化的行。
- 为超长 turn 引入虚拟化和高度缓存，控制 DOM 节点数量。

### 4.6 次要优化

以下措施有收益，但不能替代前面的根本修复：

- 开启 gzip / Brotli，减少网络字节数。
- 用 Web Worker 解析大型 JSON，降低主线程停顿。
- 调低列表模型中的 block preview 上限，完整内容按需加载。
- 页面隐藏时暂停非必要渲染。

### 4.7 验收指标

- 任意时刻进行中的 `/api/conversations` 请求不超过 1 个。
- 高频 Trace 下不形成持续增长的请求队列。
- 版本未变化时不下载完整模型，也不触发会话 DOM 刷新。
- 首屏可以先显示连接状态和会话摘要，不等待完整历史详情。
- 典型会话的首屏接口响应从几十 MB 降到 MB 以下，摘要接口目标控制在数百 KB 内。
- 连续运行 30 分钟后，浏览器内存不因刷新请求堆积而持续增长。
- turn / item 单击快速跳转和时间显示交互保持不变。

### 4.8 实施状态

已完成本轮目标，按“禁止重入和版本短路 -> 摘要/详情拆分 -> thread 级增量更新和局部渲染”的顺序落地。

已实施：

- `/api/conversations` 请求改为 single-flight，同一时刻最多一个模型请求。
- 请求期间的新事件合并为一次 dirty 补刷，不积累刷新队列。
- 前台刷新间隔调整为 1 秒；页面隐藏时调整为 5 秒。
- 暂停状态不再继续安排会话刷新，恢复时立即补刷。
- 增加版本短路，版本相同时返回轻量 `unchanged` 响应。
- 新增 `/api/conversations/summary`，只返回 session、thread、turn 摘要。
- 新增 `/api/conversations/thread?threadId=...`，只加载当前 thread 的 blocks 和详情。
- 首次加载先获取摘要并显示页面，再异步补齐当前 thread。
- 为 thread 增加独立版本号；未变化的已加载 thread 直接复用。
- session 列表按签名复用未变化的 DOM 行，不再每次全部重建。
- 启用内容筛选时自动回退完整模型，清空筛选后恢复摘要模式。
- conversation 接口增加 `Server-Timing`、响应类型和响应字节数响应头。
- `debugPerf=1` 时记录模型请求数、最大并发、unchanged 次数和响应大小。

高频验证：

- 一次注入 100 条 SSE 事件。
- 前端只新增 1 次摘要请求和 1 次当前 thread 请求。
- `modelMaxActive = 1`，未发生模型请求重入。
- 无变化刷新响应约 48 bytes。
- 内置浏览器已验证会话切换、turn 单击跳转、item 单击跳转和时间显示正常。

正式端口 45123 重启后的当前数据实测：

```text
会话摘要：       236,086 bytes，构建约 2.4ms，JSON 序列化约 2.1ms
当前最大 thread：15,853,772 bytes，完整下载约 1.54s
```

当前最大 thread 本身仍然较大，但它只在选中或发生变化时单独请求，并且不会与另一个模型请求重入。后续 block / item 级 changes 接口可以继续降低这 15MB 的详情更新成本。

后续仍可继续细化到 block / item 级 changes 接口和 turn 虚拟化，但不属于本轮止血与首屏减载的必要条件。
